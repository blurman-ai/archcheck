# Прогон #054: архитектурный дрейф C++-проектов по истории

## TL;DR (выжимка)

- **Что построили (archcheck):** CLI-инструмент, который по дереву исходников
  C++ (или `compile_commands.json`) строит граф зависимостей и считает
  архитектурные метрики — **дублирование кода** и **циклы/связность графа**.
  Работает **zero-config**: без файла настроек, на дефолтных правилах и фильтрах.
- **Что запустили и на чём:** прогнали обе метрики по **истории** 85 реальных
  C++-проектов (по 25 точек на проект), отслеживая изменение метрик от первого
  коммита к последнему.
- **Что выяснилось:** дублирование **растёт у 39 из 85** проектов (медиана прироста
  +8 п.п., максимум +69 п.п.), **снижается у 18** — значит дрейф реален, но
  **обратим**. Циклы в графе появились у 15 проектов. Дублирование и граф портятся
  **независимо** друг от друга.
- **Основной вывод:** архитектура деградирует **в конкретных коммитах**, и эти
  коммиты находятся автоматически. Пример: один «косметический» рефактор в acts
  (CERN) добавил по строке `#include` в 70 файлов — и создал **40 циклических
  зависимостей за раз**. На ревью диф выглядит безобидно; archcheck его ловит.

> **Глоссарий метрик** (дальше в тексте используются без расшифровки):
> - **dup / дублирование** — доля строк кода в дублированных блоках (%).
> - **cross-file** — число дублированных блоков, разнесённых по РАЗНЫМ файлам
>   (копипаст между компонентами, а не внутри одного файла).
> - **nodes (узлы)** — компоненты (файлы) в графе включений.
> - **edges (рёбра)** — зависимости-включения между компонентами.
> - **edges/nodes (связность)** — среднее число зависимостей на компонент; рост =
>   усиление сцепления (coupling).
> - **цикл (sccs_cyclic)** — число циклических групп зависимостей (A→B→A). Циклы —
>   главный «запах» плохой физической структуры (по Lakos).
> - **largest_scc** — размер крупнейшей такой группы (сколько компонентов в одном
>   клубке).
> - **снапшот** — состояние репозитория на одном из выбранных коммитов истории.

**Даты прогонов:** 2026-05-30 / 05-31. **Корпус:** AIDev (HuggingFace
`hao-li/AIDev`, CC BY 4.0) — C++-проекты, где использовались AI-агенты.
**Режим:** zero-config (дефолтные правила и фильтры, без `.archcheck.yml`; шум
можно было бы дополнительно давить конфигурацией, но мы этого НЕ делали).
**Инструменты:** дублирование (`line_duplication`), граф (`archcheck --graph`),
partial/token-дубли (`partial_duplication`).

> Все числа — прямой вывод инструментов на реальных репозиториях. Артефакты
> помечены и объяснены, а не замолчаны.

## 0. Зачем (переформулировка вопроса)

Исходный вопрос (#033) — «эродирует ли ИИ архитектуру» — на статическом срезе
include-графа дал почти ноль дрейфа. Эта сессия сменила постановку: **не «кто
автор коммита», а «в каком коммите архитектура поехала»** — измеряем дельту метрик
по истории. Автор (ИИ/человек) сознательно вынесен за скобки; меряем само событие
дрейфа в графе и в копипасте.

снапшотов по **first-parent** магистрали (`history_drift.sh`), на каждом — line-dup

| Метрика | 2024-05 | 2026-05 (пик) | Динамика |
|---|---|---|---|
| duplication ratio | 12.6% | 27.6% (пик 31.1%) | **×2.2** |
| cross-file блоки | 198 | 748 | ×3.8 |
| **sccs_cyclic (циклы)** | 0 | 2 | родились **2025-04** |
| **largest_scc** | 1 | 13–15 | крупнейший цикл — до 15 компонент |
| edges / nodes | 146 / 126 | 1554 / 494 | рёбра ×10 при узлах ×4 |

**Выводы:**
- Дрейф измерим и **локализуем до коммита**. Скан 76 first-parent коммитов окна
  2025-04-03..2025-04-30 нашёл точку рождения цикла: **`d3e1c13a` (2025-04-18,

- Рёбра растут (×10) быстрее узлов (×4) → средняя связность повышается (coupling).
- Копипаст удвоился и вышел на плато ~28–31%.
- В отличие от статического среза #033, **по истории дрейф виден чётко.** Срез

## 2. Per-commit копипаст-дрейф — diff-режим #056 (pilot #1b)

Задача #056 (partial/Type-3 near-dup, token-LCS) расширена **diff-режимом**:
`--diff <sha>|<A>..<B> --repo <path>`. Для коммита C: added-фрагменты (по hunk'ам
`git diff -U0 P..C`) сверяются с деревом родителя P (`git archive`) через
bag-recall + LCS-confirm. Ловит «коммит скопировал существующий код с правками» =
рождение missing-reuse-edge именно в этом коммите. Snapshot-режим не сломан.

Серия: `runs_history/optiscaler_partial/` (30 из 4063 коммитов, та же выборка;
`covered=30 skipped=4033` — без молчаливого усечения).

**Headline: 5 из 30 sampled-коммитов внесли partial-дубли.**

### Проверено вручную (независимо от агента)

Самый чистый случай — `e0189ee8` «Added XeSS Dx11 inputs»: новый файл
`inputs/XeSS_Dx11.cpp` целиком скопирован из `inputs/XeSS_Dx12.cpp` (lcs=1.0),
отличие — `_d3d12InitParams` → `_d3d11InitParams` (копия файла под новый вариант
API; см. §3bis пример 3). Ещё `469ca977` — Win32-хуки `hkLoadLibrary*`,
размноженные копипастом внутри `dllmain.cpp`.

**Оговорки (честно):** recall — связывающее ограничение (rare-token индекс
отбрасывает сильно дивергированные копии → числа это пол, не потолок);
`added_frags=0` у многих sampled-коммитов нормально (конкретный коммит часто не
добавляет код функционального масштаба); плотный скан нашёл бы больше событий.

## 3. Дисциплина vs звёзды (pilot #2)

Гипотеза: «меньше звёзд → меньше ревью → хуже дисциплина». Стратифицированная
выборка по звёздам + whole-tree line-dup на каждой репе. **n=245** (83×≥100★ +
160 низко-звёздных, по 40 на бакет). `discipline.tsv` (промежуточный n=130 —
`discipline_n130.tsv`). Метрика — median ratio (устойчива к выбросам), sig_loc≥200.

| Бакет ★ | n | median | mean |
|---|---|---|---|
| 0–2 | 34 | **6.8%** | 10.2% |
| 2–10 | 39 | 12.4% | 17.3% |
| 10–30 | 39 | 12.9% | 14.6% |
| 30–100 | 40 | 10.9% | 13.3% |
| 100+ | 83 | 11.1% | 16.7% |

**Вывод: гипотеза НЕ подтверждается** (на удвоенной выборке — ещё убедительнее).
Медиана дублирования плоская (7–13%) по всем бакетам, монотонного градиента по
звёздам нет. Самый низко-звёздный бакет [0,2) — с **наименьшей** медианой (6.8%),
прямо против гипотезы. **Звёзды не предсказывают копипаст-дисциплину.** (На n=130
картина была та же — вывод устойчив к росту выборки.)

### Аномалии (выборочно проверены)

- **ggml-org-central** (0★, 80%, sig=389k) — не репа, а **мета-агрегатор**: на

## 3bis. Проверяемые примеры (SHA + file:line + как перепроверить)

> Все примеры на репе `~/oss/_aidev_run/OptiScaler` (если не указано
> иное). Команды можно выполнить дословно. Все сниппеты ниже — реальный вывод
> `git show`, не пересказ.

### Пример 1 — рождение цикла в графе (граф-дрейф)

**Коммит:** `d3e1c13a47...` (2025-04-18, «Merge branch 'refactor-hooks'»),
parent `78b7e7c316832da502039fcb41120332bef5b8ad`.

Перепроверка (worktree на parent и на commit, сравнить `sccs_cyclic`):
```bash
cd ~/oss/_aidev_run/OptiScaler
AC=~/projects/cpparch/build/debug/src/archcheck
git worktree add --detach /tmp/chk 78b7e7c3 && "$AC" --graph /tmp/chk/OptiScaler | grep sccs_cyclic   # -> 0
git -C /tmp/chk checkout --detach d3e1c13a && "$AC" --graph /tmp/chk/OptiScaler | grep sccs_cyclic      # -> 2
git worktree remove --force /tmp/chk
# масштаб мержа:
git show --stat d3e1c13a | tail -1     # -> 81 files changed, 9272 insertions(+), 8353 deletions(-)
```
Какие рёбра замкнули цикл (added на `proxies/`):
```bash
git show d3e1c13a:OptiScaler/dllmain.cpp | wc -l   # монолит распилен; ср. с parent (3550 строк dllmain.h -> stub)
"$AC" --diff 78b7e7c3..d3e1c13a ~/oss/_aidev_run/OptiScaler/OptiScaler | grep -i proxy
```
Суть: рефакторинг распилил `dllmain.cpp/.h` на `proxies/`+`hooks/`; новые
зависимости на хаб `proxies/KernelBase_Proxy.h` замкнули 4-компонентный цикл.

### Пример 2 — параллельные реализации интерфейса под платформы (копипаст)

**sentry-unreal** (плагин краш-репортинга для Unreal; дублирование росло 3% → 16%).
Платформенные реализации одного интерфейса `ISentrySubsystem` — копии друг друга:

```bash
d=~/oss/_aidev_run/sentry-unreal
b=plugin-dev/Source/Sentry/Private
diff "$d/$b/Android/AndroidSentrySubsystem.h" "$d/$b/Apple/AppleSentrySubsystem.h"
# объявления методов класса совпадают; различается только платформенный backend
```
`FAndroidSentrySubsystem` и `FAppleSentrySubsystem` оба наследуют
`ISentrySubsystem` и объявляют идентичный набор методов (детектор: 44-строчный
cross-file блок). Это типовой «копипаст по оси платформы»: общую часть надо было
поднять в базовый класс, а различия оставить в платформенных переопределениях.
То же повторяется в `AndroidSentryEvent.h`↔`AppleSentryEvent.h`,
`...Scope.h`↔`...Scope.h` — копия размножена по всем сущностям плагина.

## 5. Сухой остаток

1. **Дрейф РЕАЛЕН и измерим — но по истории, не по срезу.** OptiScaler: копипаст
   ×2, цикл из 15 компонент, рёбра ×10. #033 не видел этого, потому что мерил
   статику.
2. **Дрейф локализуется до коммита.** Граф: коммит рождения цикла найден точно
   (OptiScaler `d3e1c13a`, acts `233ba90c`). Копипаст: diff-режим #056 указывает
   конкретный коммит-копипастер (проверено вручную).
3. **Звёзды ≠ дисциплина.** Гипотеза «меньше звёзд → хуже» не подтвердилась на 130
   репах: медиана дублирования плоская по всем бакетам.
4. **Шум-классы подтверждены на новых данных:** мета-агрегаторы и embedded
   data-blob'ы (camera_index.h) раздувают whole-tree ratio → нужен #056
   numeric-data фильтр и детект агрегаторов.

**Вердикт:** дрейф преобладает, но НЕ универсален. Копипаст растёт почти у

(циклы) у 20/85. **Копипаст и граф дрейфуют независимо** (см. контр-примеры 7.4)
— один сигнал не предсказывает другой.

### 7.1 Причина A — КОПИПАСТ-ДРЕЙФ (накопление дублей)

Топ растущих (реальный baseline > 0, ≥5 валидных снапшотов):

| repo | период | dup % | Δ |
|---|---|---|---|
| ReP_AL-3D-Lawn-Mower | 2020→2021 | 15.7 → 84.8 | +69.0 |
| OptiScaler | 2023→2026 | 4.5 → 27.6 | +23.1 |
| sentry-unreal | 2022→2026 | 3.0 → 16.0 | +13.0 |
| opentelemetry-cpp | 2020→2026 | 12.0 → 20.3 | +8.2 |
| opendht | 2015→2026 | 3.4 → 10.4 | +7.0 |
| stellar-core | 2014→2026 | 3.9 → 9.6 | +5.7 |

**Проверяемый пример — opencv-mobile** (7.6% → 41.4%, cross-file блоков 1→109,
постепенный рост 2023-2024). Топ-дубль — JPEG-энкодер, скопированный под каждую
платформу:
```bash
d=~/oss/_aidev_run/opencv-mobile
/tmp/line_dup_build/line_duplication "$d" --top 4
# [CROSS-FILE] 600 lines  highgui/src/jpeg_encoder_v4l_cix.cpp:81 <-> highgui/src/jpeg_encoder_v4l_rpi.cpp:84
diff <(sed -n '81,120p' "$d/highgui/src/jpeg_encoder_v4l_cix.cpp") \
     <(sed -n '84,123p' "$d/highgui/src/jpeg_encoder_v4l_rpi.cpp")   # -> пусто: блоки идентичны
```
600-строчный энкодер целиком скопирован `cix`↔`rpi` (а ещё `capture_v4l2.cpp` ↔
`capture_v4l2_aw_isp.cpp`, 211 строк). Механизм дрейфа: **«новая платформа =
скопировать существующий файл и поправить по месту»** вместо общей абстракции.

**Проверяемый пример — sentry-unreal** (dup 3% → 16%; см. §3bis, пример 2):
платформенные реализации интерфейса `ISentrySubsystem` (`Android…` ↔ `Apple…`) —
копии друг друга по всем сущностям плагина (Subsystem/Event/Scope). Механизм
дрейфа тот же: **«новая платформа = скопировать класс»** вместо общего базового.

### 7.2 Причина B — ГРАФ-ДРЕЙФ (циклические зависимости) ⭐ сильнейшая находка

Репы, родившие/нарастившие циклы (sccs_cyclic):

| repo | циклы 1-й→посл. | largest_scc | nodes |
|---|---|---|---|
| **acts** | **0 → 46** | 3 | 2768 |
| diagnostics | 4 → 9 | 12 | 1043 |
| itksnap | 0 → 4 | 2 | 726 |
| firmware | 0 → 4 | 3 | 824 |
| alpaka | 0 → 3 | 3 | 878 |
| FastLED | 1 → 3 | **47** | 2006 |

**Флагманский пример — acts (ACTS, CERN tracking software):** циклы стояли на 0-3
**пять лет** (2020-2025), затем **рывок 3 → 43 в ОДНОМ коммите**.

Локализация (бинарный поиск по first-parent):
```bash
d=~/oss/_aidev_run/acts
AC=~/projects/cpparch/build/debug/src/archcheck
git -C "$d" worktree add /tmp/w 233ba90c^ && "$AC" --graph /tmp/w | grep sccs_cyclic  # -> 3
git -C /tmp/w checkout 233ba90c          && "$AC" --graph /tmp/w | grep sccs_cyclic  # -> 43
git -C "$d" worktree remove --force /tmp/w
git -C "$d" show -s --format='%H %as %s' 233ba90c
# 233ba90c0886... 2025-06-02  refactor: Have `.ipp` files include their `.hpp` file (#4291)
git -C "$d" show --stat 233ba90c | tail -1   # 70 files changed, 214 insertions(+), 130 deletions(-)
```
**Коммит `233ba90c` (2025-06-02): «refactor: Have `.ipp` files include their
`.hpp` file».** Это «косметический» рефакторинг на +214 строк, добавивший
`#include "X.hpp"` в каждый `X.ipp`. Поскольку `X.hpp` исторически инклюдит
`X.ipp` в конце (стандартный шаблонный паттерн), это **замкнуло взаимный include
`hpp↔ipp` в 40 двухкомпонентных циклов разом**:
```bash
git -C "$d" show 233ba90c | grep '^\+#include' | head
# +#include "Acts/Clusterization/Clusterization.hpp"   (добавлено в Clusterization.ipp)
# +#include "Acts/EventData/Seed.hpp"                   (добавлено в Seed.ipp)  ...
```
Вывод: **рефакторинг, сделанный ради «правильности», обрушил граф зависимостей.**
Именно такой дрейф archcheck ловит, а ревью пропускает (диф выглядит безобидно:
+1 инклюд на файл).

**Второй пример — OptiScaler `d3e1c13a`** (см. §1): мерж «refactor-hooks» (81
файл, +9272/−8353) распилил монолит `dllmain` на `proxies/`+`hooks/` и замкнул
4-компонентный цикл (sccs 0→2).

### 7.3 Причина C — РОСТ СВЯЗНОСТИ (coupling: edges/nodes ↑)

Среднее число зависимостей на компонент растёт — система плотнее сцеплена:

| repo | edges/nodes 1-й → посл. | nodes_last | edges_last |
|---|---|---|---|
| stellar-core | 1.9 → 6.3 | 728 | 4611 |
| opentelemetry-cpp | 2.1 → 5.7 | 1134 | 6471 |
| FastLED | 1.3 → 4.8 | 2006 | 9556 |
| bitcoin | 2.3 → 5.6 | 1510 | 8474 |
| acts | 3.1 → 5.4 | 2768 | 14990 |

Перепроверка любой репы:
```bash
~/projects/cpparch/build/debug/src/archcheck --graph ~/oss/_aidev_run/stellar-core
# edges/nodes на HEAD; ранние снапшоты — в runs_history/all/stellar-core.tsv
```
Это «тихий» дрейф: ни одного цикла, копипаст низкий, но связность утроилась —
каждое изменение задевает всё больше кода.

### 7.4 Контр-примеры — дрейф ОБРАТИМ (важный честный вывод)

Репы, где копипаст СНИЗИЛСЯ (рефакторинг победил):

| repo | dup % 1-й → посл. | циклы |
|---|---|---|
| azure-sdk-for-cpp | 41.5 → 31.4 | 0→1 |
| FastLED | 30.9 → 24.6 | 1→3 |
| oneDAL | 39.7 → 33.6 | 4→3 |
| Nidhogg | 23.1 → 16.8 | 0→1 |
| **alpaka** | **40 → 19** (через пик) | 0→3 |

**alpaka — ключевой кейс «независимости сигналов»** (`runs_history/all/alpaka.tsv`):
копипаст вырос 21%→40% (2014-2019), затем **снижен до 19%** рефакторингом — НО в
тот же период **родились циклы (0→3) и edges выросли ×100** (33→3379). То есть
команда победила копипаст, но граф продолжил усложняться. **Два вида дрейфа
независимы: дедупликация не лечит связность.** Это прямой аргумент, зачем нужны
ОБА чека archcheck (дубли + граф), а не один.

### 7.5 Исключённые артефакты (НЕ засчитаны как дрейф)

Для честности — эти высокие/растущие числа НЕ дрейф:

- **CnC_Generals_Zero_Hour** (91%) и **GacUI** (70%) — изначально высокий dup
  (fork-in-repo `Generals/`+`GeneralsMD/`; codegen). Стабильны во времени — не рост.
- **idx=0 каждой репы** = первый коммит (README/build, 0 исходников → dup=0,
  nodes=0). В сводке `drift_summary_v2.tsv` baseline берётся с первого снапшота,
  где есть код (nodes>0 ∨ dup>0), а НЕ с idx=0. Для нескольких реп (OpenJij,
  QOpenHD, ice) даже 2-й снапшот имеет <10 узлов → их «рост 0→27%» завышен,
  помечены отдельно, в топы причин не вынесены.
- **ReP_AL** (+69%) — частично реальный (копии ESP32-скетчей), частично data-blob
  `camera_index.h` (546 строк hex, класс шума #056). В топе оставлен с пометкой.
- **SDRPlusPlusBrown** (19.8→0) и **GSL** (12.5→0) в контр-примерах — это смена
  структуры/header-only без расширений, не «починка». Отмечено.

### 7.5bis Чистка корпуса на диске (2026-05-31)

После прогона клоны, на которых archcheck **ничего не нашёл** (дубли плоско,
циклов нет, связность не растёт, абс. dup низкий) или нашёл **только ложное**

low-star 87 / ~3.6 ГБ; `~/oss` 26→20 ГБ). Полный список с причинами
и размерами — [REMOVED.md](REMOVED.md); классификация воспроизводима
(`classify_remove.py` по `runs_history/*.tsv` → `do_remove.py`).
**Сами сводки и серии (`runs_history/*.tsv`, `drift_summary*`) НЕ тронуты** — все
цифры этого отчёта остаются проверяемыми; удалены только сырые клоны (re-clonable).
Флагманские примеры (acts, OptiScaler, opencv-mobile, ncnn, alpaka, FastLED,
stellar-core, opentelemetry-cpp, diagnostics, sentry-unreal, alien, oneDAL, mc2,

### 7.6 Методологические оговорки (для воспроизводимости статьи)

1. **Сэмплинг — first-parent**, 25 снапшотов/репа. Это магистраль, не все коммиты
   (боковые ветки давали ложные провалы — см. §1). Локализация виновника — потом
   бинарным поиском между снапшотами по полной first-parent цепи.
2. **Порядок first-parent ≠ строгая хронология** (мержи возвращают старые даты).
   В сводке v2 снапшоты сортируются ПО ДАТЕ перед расчётом дельты начало→конец.
3. **Граф = include-граф** (archcheck fast backend, без libclang). Вендоринг в
   него почти не попадает (внешние инклюды не резолвятся), но subpath берётся
   автоматически (`src/` если есть код, иначе корень + дефолтные excludes).
4. **dup = line-based Type-1** (#053). Платформенные/версионные копии он видит;
   дивергированные (Type-3) — отдельный partial-проход #056 (diff-режим).
5. Все коммиты-виновники в §7.1-7.2 проверены `git show` на реальном коде.

## 8. Коммиты дрейфа: ИИ или человек? ⭐ (главный сюрприз)

> Полный разбор с текстами коммитов: `DRIFT_COMMITS_AUTHORSHIP.md`.
> Для каждого коммита-виновника смотрели по локальному git: author/committer,
> AI-trailer'ы (`Co-Authored-By: Claude/Copilot/Cursor/Devin`), бот-аккаунты,
> стиль сообщения.

Разобрано **11 коммитов-виновников** по 10 проектам.

**Вердикт: ИИ = 0, человек = 9, неясно = 2.**

**Ни один** коммит, внёсший дрейф, не имеет признаков ИИ-авторства: ноль
AI-trailer'ов, ноль бот-коммиттеров. Единственный AI-след (acts) — это
аннотация ревью `## Summary by CodeRabbit` в теле, а не подпись автора кода
(автор — человек Andreas Stefl).

**Парадокс корпуса (проверено лично):** проекты заведомо ИИ-ассистированы на
уровне репозитория —
```bash
git -C ~/oss/_aidev_run/FastLED log --all --grep='Co-Authored-By: Claude' --oneline | wc -l   # 116
git -C ~/oss/_aidev_run/firmware log --all --grep='Copilot' --oneline | wc -l                  # 176
```
— но **конкретные коммиты, ломающие архитектуру, всегда «чистые»** (без
AI-маркеров). А OptiScaler, давший сильнейший копипаст-дрейф (×5, см. §7.1),
вообще **не содержит AI-trailer'ов** во всей истории.

**Что это значит (ответ на исходный вопрос «где грязные ИИ-репы»):**
1. **По git нельзя отделить ИИ-код от человеческого** — даже там, где ИИ точно
   применяли. Trailer'ы ставятся непоследовательно; ассистент в IDE/CLI следа в
   коммите не оставляет. Атрибуция «этот дрейф сделал ИИ» по метаданным
   **невозможна** — и это честный отрицательный результат.
2. Поэтому верный угол для archcheck — **не «кто написал», а «какой коммит
   сломал»**: метрика дрейфа находит виновника независимо от авторства. Это и
   есть позиционирование инструмента.
3. **4 из 11 виновников — не «плохой код», а вендоринг/генерация:** codegen
   semconv (opentelemetry-cpp `148cfe9e`), вендоринг sqlite (stellar-core
   `ae606b93`), порт чужого репозитория (diagnostics `6629b442`), вендоренная
   JSON-либа (firmware `30d7f188`). Их в «дрейф по вине разработчика» не
   засчитываем — помечены в `DRIFT_COMMITS_AUTHORSHIP.md`.

**Характерные примеры (тексты — в `DRIFT_COMMITS_AUTHORSHIP.md`):**
- **acts `233ba90c`** — человек + CodeRabbit-аннотация. Тело прямо признаёт, что
  «.hpp инклюдит .ipp в конце» — автор знал про взаимный include и всё равно
  замкнул 40 циклов.
- **sentry-unreal `12822e19`** «Adopt generic variant type…» — список из 11
  пунктов с опечатками («funtions», «workeround», «FIx») → явный человек, LLM так
  не пишет.
- **FastLED `fab8f476`** — Conventional Commits в репе с массовым ИИ, но сам
  коммит без trailer'а → неясно.

## 9. Чем люди УБИРАЛИ копипаст (приёмы дедупликации)

> Полный разбор с фрагментами кода: `DEDUP_TECHNIQUES.md`. Анализ проектов, где
> дублирование снизилось (§7.4): какими инженерными приёмами разработчики чинили
> копипаст. Разобрано 8 проектов (7 с реальной дедупликацией, 1 артефакт).

**Топ приёмов** (13 засчитанных коммитов):

| Приём | Случаев | Суть |
|---|---|---|
| **Вынос в общий заголовок/модуль** | 5 | общий код вниз по уровням (ядро Lakos) |
| Базовый класс / инкапсуляция | 3 | вынести общее в base, различия в наследников |
| Шаблонизация / merge specialization | 2 | `Foo<bool TBlocking>` вместо двух копий |
| Извлечение функции | 2 | повторяемый блок → одна функция |
| Удаление мёртвых копий | 1 | выпилить устаревшие дублированные алгоритмы |

Сквозной мотив: **обобщение копий вдоль «осей вариативности»** — платформы
(CUDA/HIP, Android/Apple), графические бэкенды (DX/GL/Metal/Vulkan), типы.

**Яркие примеры (проверены `git show`):**
- **acts `e4a3ec90a`** — три бенчмарка степпера по ~126 строк копипаста → класс
  `BenchmarkStepper` в общем заголовке; каждый файл стал ~10 строк (−343/+209).
- **alpaka `7559f91c`** — `QueueUniformCudaHipRtBlocking` + `…NonBlocking` → один
  `QueueUniformCudaHipRt<bool TBlocking>` + алиасы (шаблон вместо двух копий).
- **Effekseer `d1fece84`/`5564dccb`** — vertex-буферы из 5 рендер-бэкендов
  подняты в `EffekseerRendererCommon` (−2557/+1246, ~70 файлов).

**Главный вывод для archcheck:** дрейф лечится **конкретными физическими
приёмами** (вынос вверх по уровням, базовый класс, шаблон) — ровно тем
словарём, на котором инструмент и говорит (Lakos physical design). То есть
archcheck не только указывает «здесь копипаст», но и подсказывает направление
рефакторинга.
