# Как искать репы с плотной агентской разработкой — признаки и запрос

## ПОРОГ ОТБОРА (закрыто 2026-05-31)

**Репа берётся в исследование, если `> 50%` её коммитов после мая 2025 —
агентские.**

- Метрика — по **числу коммитов** (не LOC, не вся история).
- Окно — **commit/merge ≥ 2025-05-01** (агентская эра).
- Знаменатель — все коммиты репы в этом окне; числитель — агентские (по
  git-metadata: bot-авторы, `Co-authored-by`, agent-emails, либо AIDev PR-метки).
- Смысл: репа, ставшая в агентскую эру **преимущественно ИИ-овой** — чистый
  объект для измерения ИИ-дрейфа (мало человеческого разбавления).
- Цель набора: **≥ 100 таких реп** (источник AIDev даёт мало → добирать через
  GitHub commit-search по AI-маркерам).


**Дата:** 2026-05-31. Метод выработан на скачанных PR-таблицах AIDev
(`/tmp/aidev_pr/`), все числа — реальный вывод pandas (read-only, без скачивания
самих реп). Источник меток ИИ — `pull_request.parquet` (ground truth AIDev:
колонка `agent`).

## Проблема

«Плотная агентская разработка» ≠ «много AI-PR». Зрелый проект с хорошим ревью
**медленно** мержит PR (очередь на review), и счётчик PR обманывает: shader-slang
имеет 17 AI-PR, но медиана merge latency 23.8ч и НИ ОДНОГО влитого за час — там
ИИ-вклад фильтруется. И наоборот, MQSensorsLib влил все 9 AI-PR мгновенно (0.0ч) —
вот это плотная агентская: код проходит как есть, без барьера ревью.

→ **главный признак — НЕ объём PR, а СКОРОСТЬ их мержа** (merge latency).
Подтверждено и литературой: «число итераций ревью до принятия» называют сигналом
качества AI-кода (Moderne; exceeds.ai). Latency — операционализация того же.

## Признаки (проверены на C++-срезе AIDev, май-2025+)

| # | Признак | Параметр запроса | Порог (эмпирич.) | Почему |
|---|---|---|---|---|
| 1 | **Окно эры агентов** | `merged_at ≥ 2025-05-01` | — | до мая-2025 — «доагентский» код |
| 2 | **Merge latency (ГЛАВНЫЙ)** | `median(merged−created)` | `< 6ч` (плотн.), `< 1ч` (оч. плотн.) | быстрый мерж = слабое ревью = ИИ как есть |
| 3 | Доля мгновенных | `% PR с latency < 1ч` | высокая | поток без барьера |
| 4 | Объём AI-PR | `count(merged AI-PR)` | `≥ 3` (для статистики) | иначе нет выборки |
| 5 | Объём AI-кода | `sum(additions)` по PR | — | строки важнее числа PR (1 крупный PR > 10 мелких) |
| 6 | Не-флагман | `stars` (грубый прокси) | низкие/средние | у флагманов ревью вычищает ИИ |
| 7 | Содержательность | feat+fix доля (`pr_task_type`) | — | реальный код, не docs/chore |

Распределения по C++ (208 merged AI-PR, май-2025+): медиана latency **2.6ч**,
**90/208 влиты < 1 часа**, 153/208 < суток. То есть быстрый мерж — норма для
агентских PR, а медленный (>17ч) — маркер review-барьера.

## Параметризованный запрос (на PR-таблицах AIDev)

```
ВХОД: pull_request.parquet (AI, ground truth), all_repository.parquet (язык/звёзды),
      pr_commit_details.parquet (LOC), pr_task_type.parquet (тип)
ФИЛЬТР:
  language == 'C++'                          # язык
  merged_at >= 2025-05-01                    # окно эры агентов
  merged_at IS NOT NULL                      # только влитые
АГРЕГАЦИЯ на репу:
  merged       = count(PR)
  med_lat_h    = median((merged_at-created_at) в часах)   # ГЛАВНЫЙ
  fast1h       = count(lat < 1ч)
  ai_loc       = sum(additions из pr_commit_details по pr_id)
  featfix      = доля type in (feat,fix) из pr_task_type
ОТБОР кандидата плотной агентской:
  merged >= 3  AND  med_lat_h < 6  AND  ai_loc >= 300
СОРТИРОВКА: med_lat_h ASC (самые «непроверяемые» сверху)
```

z3 (5.1ч). Их высокий счётчик PR обманчив.

## Оговорки

- **C++ — тонкий срез AIDev:** 33596 агентских PR всего, в C++ только 208 (66 реп).
  Это ограничивает выборку — плотных C++ реп объективно мало.
- **`human_pull_request` НЕ полный базлайн** (6618 строк / 818 реп, разрежён) —
  человеческую сторону для within-repo контраста брать из git-истории.
- **Звёзды — грубый прокси ревью**, не идеальный (бывают популярные репы со слабым
  ревью и наоборот). Latency надёжнее.
- Латентность можно подделать (squash-merge ботом сразу), но в связке с объёмом
  кода и feat/fix-долей даёт устойчивый сигнал.

## Связь с исследованием

Эти признаки → shortlist реп для within-repo теста ИИ-дрейфа (#054). Гипотеза,
которую можно проверить ровно на «плотных» репах: дрейф (дубли+граф), вносимый
AI-PR, выше человеческого — потому что слабое ревью (низкая latency) не ловит его.
