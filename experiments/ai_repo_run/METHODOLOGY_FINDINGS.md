# Как искать C++-репы с плотной агентской разработкой — рабочий метод

**Дата:** 2026-05-31. Метод выработан и валидирован вживую (`gh`, реальные клоны).
Цель — научиться надёжно находить репы, где **> 50% коммитов после мая 2025 —
агентские** (порог закрыт пользователем, см. [DENSITY_SIGNALS.md](DENSITY_SIGNALS.md)),
и набрать ≥100 таких реп.

## Вывод №1 — чем НЕ надо искать: commit-search

`gh search commits "Generated with Claude Code" committer-date:>2025-05-01` для
**широты бесполезен**. Индекс commit-search кластеризует выдачу: из 100 коммитов
получаются 2–3 уникальные репы, одна гиперактивная репа даёт 62 из 100
(`kyasbalme/Scrapbox`). Сортировка по дате не спасает. Плюс нет квалификатора
`language` и быстро ловится secondary rate-limit (403).

→ commit-search годится только чтобы **подсветить уже плотные vibe-репы**
(высокая концентрация в одной репе), но не перечислить много реп.

## Вывод №2 — чем НАДО искать: PR-search по bot-автору

`gh search prs` **поддерживает `--language=cpp` и `--author=app/<bot>`
одновременно** — язык фильтруется прямо в запросе. Это масштабируемый путь.

Две ловушки, на которые потрачено время:
- поле репы в JSON у PR-search — **`repository.nameWithOwner`**, НЕ `fullName`
  (у commit-search наоборот — `fullName`). Если перепутать — приходит пусто;
- автор-бот задаётся как `app/<login>` (например `app/copilot-swe-agent`).

Рабочая команда:
```bash
gh search prs --author='app/copilot-swe-agent' --language=cpp --merged \
  'created:>2025-05-01' --limit 1000 \
  --json repository --jq '.[].repository.nameWithOwner' | sort -u
```

## Метрика концентрации (числитель и знаменатель из одного источника)

Из bot-search получаем только «репа имеет ≥1 AI-PR» — этого мало. Концентрацию
считаем по git-истории клона:

1. **blobless-клон** — метаданные без файлов, дёшево даже для крупных реп:
   `git clone --filter=blob:none --quiet <url>`;
2. **знаменатель** — все коммиты после мая 2025:
   `git log --since=2025-05-01 --pretty=oneline | wc -l`;
3. **числитель** — AI-коммиты через **нативные фильтры git** (метод D), union по
   уникальным SHA (надёжнее, чем `grep -z` по `git log -z` — тот молча падает):
   ```bash
   { git log --since=2025-05-01 -i -E --author="$MARK"    --pretty=%H
     git log --since=2025-05-01 -i -E --committer="$MARK" --pretty=%H
     git log --since=2025-05-01 -i -E --grep="$MARK"      --pretty=%H; } | sort -u | wc -l
   ```
4. `conc% = ai*100/total`, порог **> 50%**.

**MARK** (git-метаданные: author/committer name+email + тело сообщения):
```
noreply@anthropic\.com | copilot | devin-ai | cursoragent | cursor\[bot\] |
google-labs-jules | Generated with Claude Code | Co-Authored-By: Claude |
Generated by Devin | 🤖 | swe-agent
```

## Расширение охвата: месячный разрез пробивает потолок 1000 (2026-05-31, вечер)

GitHub search отдаёт **максимум 1000 результатов на запрос** — `--limit 1000` по
боту молча обрезал выдачу. Лечится **разрезом по календарным месяцам**
(`created:YYYY-MM-01..след-месяц`), объединением уникальных реп:

- **Copilot:** один безлимитный запрос дал 230 реп; помесячно (авг-2025…апр-2026,
  ~250-300/мес) — **2037 уникальных**. Потолок резал в ~9 раз.
- Devin помесячно дал меньше безлимитного (143) — у него C++-поток редкий, разрез
  не нужен; объединил оба.
- **Мастер-пул кандидатов: 2212** уникальных C++-реп (было 410).

### Важно: какой детектор дублей используется

Триаж концентрации — git-метаданные (не зависят от чекера). А **аудит дрейфа**
(`DENSE_AUDIT_REPORT.md`) считает дубли детектором **#056 token-precise**
(selective normalization + token-LCS@0.80 + FP-фильтры), НЕ старым line-#053.
archcheck — свежая сборка. Чекеру как истине **не доверяем** → отчёт это *claims*
с `file:line`+SHA для глазной проверки, по его вердикту ничего не удаляется.
