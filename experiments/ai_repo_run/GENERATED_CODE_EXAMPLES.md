# In-tree сгенерённый код → ложный копипаст (примеры на ревью)

**Дата:** 2026-06-01. По запросу: «сгенерённый код в дереве — покажи примеры, я подумаю
что с этим делать». Это **класс FP**, который vendor-exclude (#064) НЕ ловит: файлы лежат
не под `third_party/external/vendor`, а внутри дерева проекта (`gen/`, `generated/`, рядом с кодом).

## Скан корпуса (317 реп, вне vendor-папок)

| тип | файлов | генератор |
|---|---|---|
| `*.pb.h` | 23 | protobuf |
| `*.pb.go` | 10 | protobuf (Go, для C++-детектора не важно) |
| `*.pb.cc` | **9** | protobuf — главный источник C++ FP |
| `ui_*.h` | 3 | Qt uic |

Плюс по текст-маркерам (`@generated` / `DO NOT EDIT` / `Code generated by`) — точечно в
pulp (Skia autogen под `external/` — уже отсекается), CrispASR, MPAPP, ball.

## Конкретный пример (самый яркий)

**`Ball-Lang/ball` → `cpp/shared/gen/ball/v1/ball.pb.cc` — 18 410 строк** сгенерённого protobuf.

Два механизма ложного «копипаста» одновременно:

**(1) Внутри файла** — болванка дескриптора повторяется на КАЖДОЕ сообщение дословно
(строки ~30-48), отличается только комментарием с именем типа:
```cpp
// ::ball::v1::Program
{&::_pbi::kDescriptorMethods, &::descriptor_table_ball_2fv1_2fball_2eproto, /* tracker*/ nullptr,},
// ::ball::v1::Module
{&::_pbi::kDescriptorMethods, &::descriptor_table_ball_2fv1_2fball_2eproto, /* tracker*/ nullptr,},
// ::ball::v1::ModuleAsset
{&::_pbi::kDescriptorMethods, &::descriptor_table_ball_2fv1_2fball_2eproto, /* tracker*/ nullptr,},
...
```
Детектор внутри-файла (наша цель — «скопировал и поправил часть») увидит здесь десятки
идентичных блоков. Это **ложный** сигнал: их не человек копипастил, их сгенерил protoc.

**(2) Между файлами** — серилизационный скелет (`::_pbi::`, `_InternalSerialize`,
`ByteSizeLong`, table-driven парсер) идентичен во ВСЕХ 9 `.pb.cc`. Cross-file детектор
пометит их как массовый дубль между компонентами.

## Почему #064 (vendor-exclude) не помогает

`ball.pb.cc` лежит в `cpp/shared/gen/` — это **дерево проекта**, не `third_party/`.
Текущие excludes (`third_party|external|vendor|deps|node_modules|submodules`) сюда не достают.

## Варианты решения (на твоё усмотрение)

| вариант | как | плюс / минус |
|---|---|---|
| **A. По имени файла** | skip `*.pb.cc *.pb.h *_generated.h moc_*.cpp ui_*.h qrc_*.cpp lex.yy.* *.tab.*` | просто; но не ловит нестандартные имена генераторов |
| **B. По content-маркеру** | stateful-strip файла, если в первых ~15 строках есть `@generated`/`DO NOT EDIT`/`Code generated by`/`This file was automatically generated` | надёжнее (не зависит от имени/пути); ровно как боевой SF.7 стрипает блок-комменты. **Рекомендую** |
| **C. По пути-эвристике** | skip `*/gen/* */generated/* */__generated__/*` | дёшево, ловит каталоги-помойки генераторов |
| **A+B+C** | все три слоя | defense-in-depth, минимум FP |

Мой совет: **B как основа** (маркер в шапке — самый честный признак «это не писал человек»),
A и C — дешёвые добавки. Это поведение **дедупликатора по умолчанию**, не конфиг агента.

## Важная оговорка
Реальный внутри-файловый копипаст (скопировал функцию, изменил часть) и сгенерённая болванка
по surface-метрикам похожи. Маркер/имя/путь — единственный дешёвый способ их развести ДО
семантического слоя. Без этого precision внутри-файлового прохода просядет на protobuf-тяжёлых репах.
